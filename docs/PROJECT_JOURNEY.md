# Photo Archiver 项目动机、设计与开发过程总结

## 1. 项目动机

这个项目的起点非常明确：解决“内存卡照片转档到备份盘”这件高频但容易出错的事情。

一个直接触发点是：过年外出旅游期间，相机拍了大量照片和视频，回家后手工整理非常耗时，且容易漏拷、重复拷贝和归档混乱。

在实际使用中，手工导入通常会遇到这些问题：

- **重复导入**：同一批素材多次拷贝，后续难以清理。
- **目录混乱**：不同批次、不同设备导入后，文件散落在多个目录。
- **命名冲突**：相机重复文件名（例如 `DSC_0001`）导致覆盖风险。
- **校验缺失**：拷贝完成不代表文件可靠落盘，尤其在外接盘场景下。
- **任务不可追踪**：一旦中途中断，无法快速知道哪些成功、哪些失败。
- **查找困难**：归档后无法按内容快速检索（例如“海边、人像、日落”）。

因此，项目目标不是“再写一个复制脚本”，而是构建一个**本地优先、可追踪、可扩展**的照片归档系统：

- 能稳定导入（扫描、规划、复制、校验、记录）
- 能被普通用户使用（GUI）
- 能继续演进到智能检索（内容识别、标签）

---

## 2. 核心设计目标

从一开始就围绕以下原则设计：

- **安全优先**：默认不删除源文件，复制后可校验。
- **可重复执行**：重复导入不重复入库，支持重试。
- **可观测**：每次导入任务有状态、有条目、有统计。
- **可迁移**：业务逻辑与 UI 解耦，支持后续替换前端技术栈。
- **本地优先**：所有核心能力离线运行，不依赖云服务。

---

## 3. 架构设计与技术路线

### 3.1 分层结构

项目采用“核心能力 + 表现层”分层：

- **Core（Go）**：扫描、归档、去重、校验、任务编排。
- **Storage（SQLite）**：任务与文件索引持久化。
- **API（Go Service）**：提供本地 HTTP 接口给前端。
- **UI（macOS SwiftUI）**：原生交互、任务管理、状态展示。

这种分层带来两个好处：

1. CLI / GUI / API 共用一套核心逻辑，避免重复实现。
2. 后续 UI 换代（例如 Fyne -> SwiftUI）无需重写业务层。

### 3.2 数据层设计

SQLite 作为本地状态中心，核心数据实体包括：

- `import_jobs`：导入任务（总量、成功/失败、状态、时间）
- `import_items`：任务条目（每个文件的结果）
- `files`：已归档文件索引（路径、哈希、拍摄时间等）

它支撑了以下能力：

- 去重（基于哈希）
- 失败重试（按失败条目回放）
- 任务追踪与统计
- 后续标签检索扩展

### 3.3 时间归档策略

归档时间规则采用“**EXIF 优先，文件时间回退**”：

1. 优先读取 EXIF 拍摄时间（更符合照片实际拍摄时间）
2. 若 EXIF 缺失/解析失败，回退文件修改时间

这解决了“后期拷贝导致 mtime 改变，按文件时间归档不准确”的常见问题。

## 4. UI 技术路线演进

### 4.1 早期原型：Fyne

早期使用 Fyne 快速验证流程，重点在功能可行性：

- 目录选择
- 导入执行
- 任务记录
- 设置页基础信息

Fyne 阶段的价值是“快速闭环”，但随着视觉与平台一致性要求提高，出现了限制：

- macOS 原生体验不足
- 设计语言不够统一
- 复杂交互（侧栏、状态面板）可塑性有限

### 4.2 目标落地：SwiftUI

后续切换为 SwiftUI，保持 Go Core 不变，新增本地 API 服务层。这样做到：

- UI 采用原生 macOS 体验
- 后端继续复用已有导入逻辑
- 前后端通过 localhost HTTP 解耦

并进一步实现了：

- 后端内置到 App Bundle
- 启动时自动拉起后端
- 端口冲突自动回退（38080-38090）
- 健康检查就绪后再开放页面

## 5. 关键开发过程回顾

以下按“问题 -> 方案 -> 结果”梳理本次开发过程中的关键节点。

### 5.1 从文档到仓库初始化

- **问题**：需求分散，开发入口不清晰。
- **方案**：先产出可开发文档（PRD、技术设计、DB Schema），再建仓库。
- **结果**：形成了文档驱动的开发基线，减少后续返工。

### 5.2 核心链路先打通

- **问题**：没有可用“最小闭环”。
- **方案**：先做 `scan -> plan -> import -> verify -> persist`，并提供 CLI。
- **结果**：项目具备实际导入能力，可用于真实数据验证。

### 5.3 去重与任务追踪

- **问题**：重复导入与失败不可追踪。
- **方案**：引入哈希指纹 + 任务条目落库。
- **结果**：可重复导入，失败可追溯与重试。

### 5.4 EXIF 归档正确性

- **问题**：按文件时间归档不准确。
- **方案**：EXIF 优先，mtime 回退。
- **结果**：目录更贴近拍摄真实时间线。

### 5.5 大任务超时问题

- **问题**：前端同步等待导入，长任务触发超时。
- **方案**：导入改为异步提交（返回 `job_id`），前端轮询任务状态。
- **结果**：前端不再因大任务超时，交互更稳定。

### 5.6 任务操作增强

- **新增能力**：
  - 重试失败项
  - 删除任务（非 running）
  - 中止运行中任务（cancel）
- **结果**：任务生命周期管理更加完整。

### 5.7 页面状态丢失问题

- **问题**：切换页面后导入参数和进行中任务状态丢失。
- **方案**：引入全局 Store：
  - `ImportDraftStore`（参数草稿）
  - `ImportTaskStore`（任务状态与轮询）
- **结果**：跨页面切换后任务与表单状态可保持。

### 5.8 “任务刚提交就报错”的体验问题

- **问题**：任务创建到可查询之间的短暂窗口期会显示错误。
- **方案**：增加启动宽限（初始化阶段友好提示，不立即判错）。
- **结果**：导入页体验更平滑，减少误报。

### 5.9 一键打包完整 App

- **问题**：前后端分开启动，用户体验不完整。
- **方案**：
  - 构建 Go 后端并内嵌到 `.app` 资源
  - SwiftUI 启动时自动拉起后端
  - 提供一键脚本 `scripts/build_macos_app.sh`
- **结果**：输出单个可运行 App（`dist/PhotoArchiverMac.app`）。

## 7. 已识别风险与折中

- **取消任务是“尽快停止”而非“瞬时中断”**：若正在处理单个大文件，会在当前步骤结束后停止。
- **任务详情条目存在上限**：详情展示为性能做了限制，统计应以 job 汇总字段为准。
- **本地端口冲突仍可能发生**：已做端口区间回退，但如区间都占用仍会启动失败。

---

## 8. 下一阶段建议

1. 任务详情改为分页加载（避免条目上限引发理解偏差）
2. 增加“停止后清理策略”（是否删除半拷贝文件）
3. 引入并发 worker 与限速配置
4. 推进图库与标签检索（识别结果可视化）
5. 增加发布流程（签名、公证、DMG 打包）

---

## 9. 结论

该项目已经从“导入脚本诉求”演进为“可交付的桌面应用架构”：

- 业务核心稳定（导入、校验、去重、任务状态）
- UI 体验可用且逐步原生化
- 交付链路完整（内嵌后端 + 一键打包）

更重要的是，当前架构已经具备持续迭代能力：后续无论是并发优化、识别检索，还是发布运营，都可以在现有基础上增量推进。
